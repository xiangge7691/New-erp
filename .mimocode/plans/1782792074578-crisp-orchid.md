# Plan: 审批模块问题修复

## 现状问题总结

### 问题1：审批流程引擎缺失
- 没有 `approve()` / `reject()` / `transfer()` 方法
- `currentNodeId` 从未被程序化更新
- 审批实例创建后无法推进到下一个节点

### 问题2：业务绑定不明确
- 数据库有 `approval_instance_id` 字段（purchase_orders/stock_in/stock_out）
- 但 Java 代码中没有任何业务 Service 调用审批模块
- `relatedId` + `relatedType` 模式已设计但未使用

### 问题3：回滚机制未实现
- `ApprovalNode` 有 `afterRejectStatus`（驳回后业务状态）
- 数据库有 `reject_to_node_id`（驳回到哪个节点）
- 但 Java Entity 缺少 `rejectToNodeId` 字段
- 代码中没有驳回逻辑

### 问题4：Entity 与 Schema 不同步
| 字段 | Schema | Entity | 状态 |
|------|--------|--------|------|
| approval_instance.initiator_id | ✅ | ❌ | 缺失 |
| approval_instance.is_deleted | ✅ | ❌ | 缺失 |
| approval_node.reject_to_node_id | ✅ | ❌ | 缺失 |
| approval_workflow.status | ✅ | ❌ | 缺失 |
| approval_workflow.is_deleted | ✅ | ❌ | 缺失 |
| approval_record.approved_at | ✅ | ❌ | 缺失 |

## 修改方案

### 第一步：Entity 补齐缺失字段

**ApprovalInstance.java** — 新增：
```java
private Long initiatorId;    // 发起人ID
private Integer isDeleted;   // 逻辑删除
```

**ApprovalNode.java** — 新增：
```java
private Long rejectToNodeId; // 驳回到哪个节点
```

**ApprovalWorkflow.java** — 新增：
```java
private Integer status;      // 状态：0停用/1启用
private Integer isDeleted;   // 逻辑删除
```

**ApprovalRecord.java** — 新增：
```java
private LocalDateTime approvedAt; // 审批时间
```

### 第二步：实现审批流程引擎

**ApprovalInstanceService 新增方法：**
```java
// 同意当前节点
void approve(Long instanceId, Long userId, String remark);
// 驳回
void reject(Long instanceId, Long userId, String remark);
// 转交
void transfer(Long instanceId, Long userId, Long targetUserId, String remark);
// 获取当前待处理节点
ApprovalNode getCurrentNode(Long instanceId);
```

**ApprovalInstanceServiceImpl 实现逻辑：**

#### approve（同意）：
1. 获取实例，校验状态为 PENDING
2. 获取当前节点 currentNodeId
3. 查找下一个节点（按 nodeOrder 排序，当前节点之后的第一个）
4. 记录 ApprovalRecord（action=AGREE）
5. 如果有下一个节点 → 更新 currentNodeId 为下一个节点
6. 如果没有下一个节点 → 实例状态改为 APPROVED
7. 如果节点配置了 afterPassStatus → 更新关联业务实体状态

#### reject（驳回）：
1. 获取实例，校验状态为 PENDING
2. 获取当前节点
3. 记录 ApprovalRecord（action=REJECT）
4. 如果有 rejectToNodeId → 回到指定节点，实例保持 PENDING
5. 如果没有 rejectToNodeId → 实例状态改为 REJECTED
6. 如果节点配置了 afterRejectStatus → 更新关联业务实体状态

#### transfer（转交）：
1. 获取实例，校验状态为 PENDING
2. 记录 ApprovalRecord（action=TRANSFER）
3. 实例状态改为 TRANSFERRED，记录 transferredTo

### 第三步：业务绑定机制

**统一绑定模式：**
- 审批实例：`relatedId` + `relatedType` 指向业务实体
- 业务实体：`approvalInstanceId` 指向审批实例

**新增通用方法：**
```java
// 创建审批实例并绑定业务
ApprovalInstance createWithBinding(String relatedType, Long relatedId, Long workflowId, Long initiatorId);
// 取消审批并解绑
void cancelAndUnbind(Long instanceId, Long userId, String reason);
```

### 第四步：业务侧集成（预留）

各业务 Service 在关键状态变更时调用审批模块：
- 创建采购单 → 可选创建审批实例
- 创建入库单 → 可选创建审批实例
- 审批通过 → 自动更新业务状态
