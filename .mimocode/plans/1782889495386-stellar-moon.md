# 业务单据关联设计方案

## 一、业务流程

```
工单 (WorkOrder)
  │
  ├──→ 采购单 (PurchaseOrders) ──→ 原料入库 (StockIn) ──→ 原料出库 (StockOut)
  │        [可选：有采购才有后续]        [物料领用]
  │
  └──→ 生产计划 (ProductionPlan) ──→ 成品入库 (StockIn) ──→ 成品出库 (StockOut)
           [生产执行]                   [生产完成入库]        [销售/调拨出库]
```

**说明**：
- 工单是起点，包含制剂信息和批量
- 采购单是可选的，只有需要外购原料时才有
- 原料入库/出库对应采购物料的入库和生产领用
- 生产计划是工单的核心执行环节
- 成品入库/出库对应生产完成后的入库和销售出库

## 三、方案选择：统一关联表

### 为什么不用 FK 直接加字段？
1. **可选关系**：采购单是可选的，工单不一定有采购单，加 FK 会有大量 NULL
2. **多对多**：一个工单可能对应多个采购单（不同物料），一个采购单也可能服务多个工单
3. **双向查询**：需要既能查"工单关联了哪些单据"，也能查"采购单被哪个工单引用"
4. **不侵入现有表**：不需要修改已有表结构

### 关联表设计

```sql
CREATE TABLE IF NOT EXISTS `business_document_relation` (
    `id`              BIGINT       NOT NULL AUTO_INCREMENT,
    `source_type`     VARCHAR(50)  NOT NULL COMMENT '来源单据类型',
    `source_id`       BIGINT       NOT NULL COMMENT '来源单据ID',
    `target_type`     VARCHAR(50)  NOT NULL COMMENT '目标单据类型',
    `target_id`       BIGINT       NOT NULL COMMENT '目标单据ID',
    `relation_type`   VARCHAR(50)  NOT NULL COMMENT '关系类型',
    `remark`          VARCHAR(500) DEFAULT NULL,
    `created_by`      BIGINT       DEFAULT NULL,
    `created_time`    DATETIME     DEFAULT CURRENT_TIMESTAMP,
    `is_deleted`      INT          DEFAULT 0,
    PRIMARY KEY (`id`),
    KEY `idx_source` (`source_type`, `source_id`),
    KEY `idx_target` (`target_type`, `target_id`),
    UNIQUE KEY `uk_relation` (`source_type`, `source_id`, `target_type`, `target_id`, `relation_type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务单据关联表';
```

### 单据类型枚举

| 类型代码 | 说明 |
|---------|------|
| WORK_ORDER | 工单 |
| PURCHASE_ORDER | 采购单 |
| PRODUCTION_PLAN | 生产计划 |
| STOCK_IN | 入库单 |
| STOCK_OUT | 出库单 |

### 关系类型枚举

| 关系代码 | 说明 | 示例 |
|---------|------|------|
| TRIGGER | 触发 | 工单 → 采购单 |
| DERIVE | 派生 | 工单 → 生产计划 |
| SUPPLY | 供应 | 采购单 → 原料入库 |
| CONSUME | 消耗 | 原料出库 → 生产计划 |
| PRODUCE | 产出 | 生产计划 → 成品入库 |
| SELL | 销售 | 成品出库 ← 成品入库 |

### 典型关联记录

```
# 工单 → 采购单（可选）
(work_order, 1, purchase_order, 1, TRIGGER, "采购原料")

# 工单 → 生产计划
(work_order, 1, production_plan, 1, DERIVE, "生产执行")

# 采购单 → 原料入库
(purchase_order, 1, stock_in, 1, SUPPLY, "采购入库")

# 原料出库 → 生产计划（物料领用）
(stock_out, 1, production_plan, 1, CONSUME, "生产领料")

# 生产计划 → 成品入库
(production_plan, 1, stock_in, 2, PRODUCE, "成品入库")

# 成品入库 → 成品出库
(stock_in, 2, stock_out, 1, SELL, "销售出库")
```

## 四、后端实现

### 4.1 Entity

```java
@TableName("business_document_relation")
public class BusinessDocumentRelation extends AuditEntity {
    @TableId(value = "id", type = IdType.AUTO)
    private Long id;
    private String sourceType;
    private Long sourceId;
    private String targetType;
    private Long targetId;
    private String relationType;
    private String remark;
    private Integer isDeleted;
}
```

### 4.2 Service 接口

```java
public interface BusinessDocumentRelationService extends IService<BusinessDocumentRelation> {
    // 查询某单据的所有关联单据
    List<BusinessDocumentRelation> findBySource(String sourceType, Long sourceId);
    // 查询某单据被哪些单据引用
    List<BusinessDocumentRelation> findByTarget(String targetType, Long targetId);
    // 查询完整业务链路（从工单开始的所有关联）
    List<BusinessDocumentRelation> findFullChain(String sourceType, Long sourceId);
    // 建立关联
    void createRelation(String sourceType, Long sourceId, String targetType, Long targetId, String relationType, String remark);
    // 删除关联
    void removeRelation(String sourceType, Long sourceId, String targetType, Long targetId, String relationType);
}
```

### 4.3 Controller

```
GET    /api/document-relation/source/{type}/{id}       查询某单据的关联
GET    /api/document-relation/target/{type}/{id}       查询某单据被引用
GET    /api/document-relation/chain/{type}/{id}        查询完整业务链路
POST   /api/document-relation                          建立关联
DELETE /api/document-relation                          删除关联
```
